# Accessibility Guidelines

Removery UI targets WCAG 2.1 AA as the baseline for every component. Most of the keyboard and ARIA behavior below comes for free from Radix UI, which every interactive component in this library is built on — it isn't reimplemented by hand.

## Color Contrast

All text/background pairings must meet minimum contrast ratios:

## Keyboard Navigation

All interactive components support full keyboard navigation, via Radix's built-in behavior:

- **Tab** — move focus between interactive elements
- **Enter / Space** — activate buttons, toggle switches, select items
- **Arrow keys** — navigate within composite widgets (Tabs, Menubar, Dropdown Menu, Context Menu, Navigation Menu, Radio Group, Select)
- **Escape** — close Dialog, Sheet, Drawer, Popover, Dropdown Menu, and Context Menu
- **Home / End** — jump to the first/last item in Select, Command, and menu lists

## Focus Management

- Focusable elements show a visible focus ring using `--ring` (same value as `--primary`, `#009da3` — 3.31:1 against background in both themes, clearing the 3:1 UI-component threshold)
- Focus rings use Tailwind's `focus-visible:ring-2` / `ring-[3px]` utilities, consistent across components
- Focus is trapped inside Dialog, Sheet, Drawer, and AlertDialog-style overlays (Radix `Dialog.Root` / `vaul`'s drawer both handle this)
- Focus returns to the trigger element when an overlay closes (Radix default behavior — not something this project overrides)

  ]}
/>

Removery UI doesn't have a Tree View component yet, so there's no `role="tree"` pattern to document — it'll be added here if one gets built.

## Screen Reader Support

- Use Tailwind's `sr-only` utility class for screen-reader-only text (used today in `CommandDialog`'s hidden title/description) — there's no separate `VisuallyHidden` component in this project, `sr-only` covers the same need
- Icon-only buttons (like `SidebarTrigger`) include `aria-label`
- Form inputs are associated with labels via the `Label` component's `htmlFor`
- Error messages should be linked with `aria-describedby` — this isn't wired up automatically by any component yet, it's on the implementer to add per form

## Reduced Motion

The system respects `prefers-reduced-motion`, added globally in `src/index.css`:

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

## Testing Checklist

Before shipping any new component or page:

- Verify color contrast with a tool (e.g., axe DevTools, Lighthouse) — status badges now clear 3:1 in both themes as of the 2026-07-15 sync, but `status-confirmed` in light mode still falls short of normal-text AA (see table above)
- Navigate the entire flow using only a keyboard
- Test with a screen reader (VoiceOver, NVDA)
- Confirm `prefers-reduced-motion` is respected
- Validate HTML semantics (headings, landmarks, lists)
- Check for focus trapping in overlays (Dialog, Sheet, Drawer)
- Verify touch targets are at least 44×44px on mobile — several `size="icon"` buttons in this library are currently 36px (`size-9`) and need review for touch contexts
